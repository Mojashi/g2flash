# 依頼書: Even G2 CFW へのオンデバイス3Dレンダリング機能追加

- 宛先: CFW（`patches/` 一式）担当
- 依頼者: 大森2（ステレオ表示デモ `demos/stereo-demo.ts` 作成担当）
- 対象: G2 firmware **2.2.4.34** ベースの CFW（`g2_2.2.4.34_cfw.bin`）
- 目的: グラス側で3Dモデルを直接レンダリングし、**立体地図を"ぐるっと見る"アプリ**の描画基盤にする

---

## 0. TL;DR

ホストで左右眼画像を描いて BLE でピクセルを流す現行方式は BLE 律速で数fpsが上限。**モデルを1回送り、以降は姿勢（またはオンボードIMU）だけで、グラス側が両眼フレームを毎フレーム自前レンダリングする**方式に移せば、フレームレート・タイル継ぎ目・ヘッドトラッキング遅延の3問題を同時に解消できる。既存 CFW（`zlib_glue.c` の image モードディスパッチ、`push_display`、per-lens `FW_SIDE`、osTimer、`gesture_fwd.c` のIMUイベント経路）に、新しい image モードとして無理なく載る想定。

依頼は次の2本立て:

1. **【前提バグ修正】** 単一 576×288 コンテナがリブートする件（下記 §2）。全画面ステレオにもオンデバイス3Dにも共通の前提。
2. **【新機能】** オンデバイス3Dレンダリング（下記 §3〜§6）。v1（ワイヤーフレーム）から段階導入。

---

## 1. 背景（既存アーキテクチャの前提）

実装済み CFW の関連部分（担当なら既知の想定だが、本依頼の設計はこれらに乗る）:

- **image モードディスパッチ**: `zlib_glue.c` の `image_dispatch()`。再構成済みバッファ先頭バイト = モード（`& 0x7f`）、最上位ビット `0x80` = "lenses differ"。使用中: `1,2,3,4,5,6,7,8,9` と `'B'(0x42)`。→ **`0x10` 以降が空き**。
- **ステレオ（mode 4）**: `[4][zlib(left w*h ++ right w*h)]`。`FW_SIDE()`（`0x0045a8ec`, 2=左/1=右）で各レンズが自分の半分だけを `disp` に採用。
- **描画先バッファ**: 表示バッファ A = `*(uint8_t**)(state+0x8)`（`w*h`, 8bpp）。`w=*(u16*)(state+0x40)`, `h=*(u16*)(state+0x42)`。recon バッファ B = `*(uint8_t**)(state+0xc)`。
- **パネル反映**: `push_display(state, disp, w, h)`（dcache clean → LVGL image descriptor 設定 → `FW_SETSRC` → `FW_INVAL`）。
- **両レンズ実行**: image 処理は `image_deferred`（クロスレンズ完了同期）経由で**両レンズ**で走る（`image_worker` のコメント参照）。mode 4 と同じ経路に載せれば新モードも両眼で動く。
- **永続コンテキスト**: `customCfwContext`（`getCustomCfwContext()`, magic `0xC0FFEE5A`, BLE-RXタスクの spare word `0x20003fdc` にアンカー）。モデルデータの保持先に使える。
- **自律タイマ**: mode-5 kind4 の `seq_tick`（osTimer コールバック）。`FW_TIMER_NEW(0x00448402)` / `FW_TIMER_START(0x004484ea)`、コールバックの絶対アドレスは2パス目で `SEQ_TICK_ADDR` として焼き込み。**自律レンダリングループの雛形**になる。
- **IMU 経路**: `gesture_fwd.c` に **「8 = IMU report」**（`OsEventTypeList`）。IMUは存在し報告経路もある。SysEvent 送出は `FW_SYSEVT(0x004ff232)`。
- **ビルド制約**（`build.py`）: `--target=thumbv7em-none-eabi`（**Cortex-M4/M7**）、**PIC**・外部シンボル/未解決分岐は**ハードエラー**。→ 事実上 **soft-float 不可・64bit除算(`__aeabi_uldivmod`)不可**。注入コードは整数のみ。

## 2. 【前提バグ修正】単一 576×288 コンテナのリブート

### 現象
`demos/stereo-demo.ts` を **576×288 単一コンテナ**でステレオ（mode 4）ストリームしたところ、**1フレーム目でグラスがハング→ウォッチドッグ再起動**。288×144 単一コンテナのステレオは正常表示。

### 最有力仮説（CFW実装の穴）
`patch_img_container_576.py` は `common_image_container` 生成関数（`FUN_00500f7a`）の**サイズ上限チェック（幅>0x120/高>0x90 を弾く cmp/blt）だけ**を書き換えており、コメントにも *"alloc-failure handling are untouched"*。→ **表示バッファ A（state+0x8）/ recon B（state+0xc）の確保量が旧上限 288×144（=41472B）のままで、単一コンテナが 288×144 を超えると 8bpp 全画面書き込みでバッファ溢れ → ハードフォルト → リブート**、という筋が「1枚で落ちる」挙動とよく一致。傍証: `zlib_glue.c` に "faceclaw's 4 tiles"、`buildImageContainers` ドキュメントに「576×288 は 288×144 タイルの 2×2」。＝全画面は**タイル分割が本来の想定**で、単一 576 コンテナは未運用の可能性大。

### 切り分け（1コマンドで確定）
**単一 576×288 コンテナに mode 2（非ステレオ）を1枚**送る。
- 落ちる → バッファ確保バグ確定（modeに依らない一般問題）。
- 落ちない → ステレオ固有（mode 4 のインラインデコード負荷＝別筋）を再検討。

### 依頼
- 上記切り分けの実施と結果共有。
- バッファ確保バグ確定なら、**A/B の確保量を、上限解放後の実 `w*h` に追従させる**修正（`FUN_00500f7a` 内の malloc サイズ算出、またはそれが参照する定数）。length-preserving でなければ `--recompute-checksums` 前提で。
- これは**全画面ステレオ**と**オンデバイス3Dの全画面レンダ先**の**共通前提**。ここが直れば「タイル継ぎ目同期」問題は構造的に不要になる（フレーム全体を1バッファに一括描画→1 present）。

## 3. 【新機能】オンデバイス3Dレンダリング — 設計方針

新しい image モードとして `image_dispatch()` に追加する（mode 4 と同じく `image_deferred` 経由で両レンズ実行）。各レンズは `FW_SIDE()` で自分の眼のオフセットを決め、**そのレンズの視点を直接ラスタライズ**して `push_display`。ホストは**モデルを1回**送り、以降は**姿勢のみ**（または IMU 由来でゼロ送信）。

### ステレオ投影（ホスト側 `stereo-demo.ts` と同一式・そのまま移植可）
カメラは瞳孔間距離 `IOD` だけX方向に離れた平行2眼、収束深度 `Zc` で視差ゼロ。ワールド点 `P=(X,Y,Z)`（Z>0が前方）:

```
u_L = cx + f*(X + IOD/2)/Z - f*IOD/(2*Zc)
u_R = cx + f*(X - IOD/2)/Z + f*IOD/(2*Zc)
v   = cy - f*Y/Z
```

`FW_SIDE()==2`（左）は上段、右は下段。視差は小さめ（288幅で概ね±9px以内）に保つと融像しやすい。

## 4. ワイヤ・プロトコル（提案）

先頭バイト = モード。既存慣習に合わせ、可変長ペイロードは基本 zlib 圧縮（姿勢のような極小データは非圧縮でよい）。

### mode `0x10` = MODEL_LOAD（モデル定義・置換）
`[0x10][zlib( model_blob )]`

`model_blob`（little-endian）:
```
u8   version            (=1)
u8   flags              (bit0: has_tris)
u16  nVerts
u16  nEdges
u16  nTris              (has_tris のとき有効, なければ0)
i16  verts[nVerts][3]   モデル座標（固定小数点; スケールは §6のconfig or 既定 1/256）
u16  edges[nEdges][2]   頂点インデックス対（ワイヤーフレーム用）
u16  tris[nTris][3]     頂点インデックス3つ（ソリッド用, v3〜）
u8   tri_shade[nTris]   面の明度0..255（任意, v3〜）
```
CFWコンテキストに malloc 保持（`customCfwContext` に `model_*` フィールド追加、または別malloc＋ポインタ保持）。再送で置換。上限（頂点/エッジ/三角形数）はメモリに応じて設定し超過は拒否。

### mode `0x11` = RENDER（姿勢セット＋描画）
`[0x11][pose]`（**非圧縮・~20〜30B**）
```
i16  rot[9]   or  i16 quat[4]   カメラ回転（固定小数点; どちらか実装しやすい方）
i16  trans[3]                    カメラ平行移動（モデル座標）
（f, IOD, Zc は §6 config の現行値を使用。pose に載せず config 管理を推奨）
```
処理: モデル頂点を pose で変換 → §3の式で当該レンズに投影 → `disp` を背景クリア → エッジ（v1）/ 面（v3）をラスタライズ → `push_display`。両レンズは完了同期で各自実行。

### mode `0x12` = RENDER_CTRL（設定・自律ループ）
`[0x12][sub][args...]`（mode 5/7 のサブディスパッチに倣う）
```
sub 0: 投影パラメータ設定  [i16 f][i16 iod][i16 zc][i16 near]   (固定小数点)
sub 1: 自律IMUループ開始    [u16 interval_ms]   osTimerで毎tick IMU→両レンズ描画
sub 2: 自律ループ停止
sub 3: 描画スタイル         [u8 style(0=wire,1=solid)][u8 fg][u8 bg]
```

> ホスト側（`g2-kit` / demo）の対応送信ヘルパも本依頼のスコープ外だが、確定次第こちらで実装する。プロトコルは擦り合わせさせてほしい。

## 5. IMU 連携（v2）

- `gesture_fwd.c` の「8 = IMU report」から、**IMUサンプルの取得点**（イベント8の背後の値、または下位センサ読み出し）を特定してほしい。
- 報告が**姿勢（融合済みquat/euler）**なら、それをカメラ回転に直結。**生の accel/gyro** なら、オンデバイスで**固定小数点の姿勢推定（相補フィルタ/簡易Madgwick）**が必要。
- 実現すれば「**首を振ると視点が動く**」が**ゼロBLE遅延**で成立（立体地図の核心）。
- **要検討（スレッド安全）**: 自律ループを osTimer コールバック（`seq_tick`同様、タイマスレッド）から回す場合、LVGL/表示バッファ操作が表示スレッドと競合しうる。安全なら image_deferred と同経路にディスパッチする、UIイベントを介す等、既存の描画スレッド上で回す設計を推奨。ここは実装判断に委ねる。

---

## 6. 実装制約・メモリ（重要）

- **固定小数点必須**（Q16.16 目安）。`build.py` の PIC 制約で **float ランタイム（`__aeabi_f*`）も 64bit除算（`__aeabi_uldivmod`）も外部シンボル＝ビルド不可**。透視除算は 32bit ハードウェア `UDIV`、32×32→64 は `UMULL`（インライン）に留める。
  - 代替: チップが **M4F** なら `build.py` に `-mfpu=fpv4-sp-d16 -mfloat-abi=hard` を足せば VFP 命令がインライン化されハードfloatも可（別途 FPU 電源/CPACR 有効化の確認要）。**FPU有無の確認**を依頼したい。どちらでも設計は成立。
- **描画先** = 表示バッファ A（`state+0x8`, `w*h` 8bpp）。全画面（576×288=165888B/眼）で回すには **§2のバッファ修正が前提**。まずは 288×144 で v1 を通し、§2完了後に全画面へ。
- **ワイヤーフレーム**: Zバッファ不要。背景クリア＋固定小数点 Bresenham/Wu 線。M4 @ ~200MHz なら数千頂点＋線描画で両眼 30–60fps は十分射程。
- **ソリッド（v3）**: Zバッファ `w*h`（16bit → 2*w*h）を両レンズで再利用（左描画→push→クリア→右描画→push）。全画面Zバッファ ~331KB のヒープ予算確認要。フィルレート次第で数十fps。
- 演算量目安（M4 @ ~200MHz）: 頂点変換 数千でも <1ms、線描画 ~1ms。実質律速は LVGL/パネル反映側になる見込み。

## 7. 段階マイルストーン

| 版 | 内容 | 依存 |
|----|------|------|
| **v1** | オンデバイス**ワイヤーフレーム**描画。mode `0x10` MODEL_LOAD + `0x11` RENDER（姿勢はホスト送信）。まず 288×144 で実証 | §1既存基盤のみ |
| **v2** | **IMU駆動**の姿勢（mode `0x12` sub1/2）。ヘッドトラッキング見回し | §5 IMU取得点特定 |
| **v3** | **ソリッド＋シェーディング＋Zバッファ** | メモリ予算確認 |
| **v4** | **地図コンテンツ**（OSM建物押し出しメッシュ）を MODEL_LOAD → 立体地図をぐるっと | v1〜v3, §2全画面化 |

v1 が費用対効果最大。まずここを合意したい。

## 8. 要確認事項（担当への質問）

1. §2 切り分け（単一576に mode2 を1枚）の結果。バッファ確保バグか否か。
2. チップは M4F か（**FPU有無**）。→ 固定小数点 or VFP の判断。
3. 「IMU report（イベント8）」の中身は融合姿勢か生センサか。取得点（RAMアドレス/関数）はどこか。
4. モデル/Zバッファ用に使えるヒープ予算の目安（全画面Zバッファ ~331KB は許容範囲か）。
5. 自律ループを回す安全なスレッド/経路の推奨（osTimer直 vs 描画スレッド経由）。

## 9. 成果物・受け入れ基準

- **前提修正**: 単一 576×288 コンテナに 8bpp 全画面（mode2/4）を送ってもリブートしない。
- **v1**: `[0x10]` でモデル投入後、`[0x11]` の姿勢送信ごとに両眼ワイヤーフレームが更新され、ホストからの姿勢変化に滑らかに追従（288×144で目視確認）。左右で正しい視差（融像時に立体視できる）。
- **v2**: 頭の回転に対し体感遅延の小さい見回しが成立。
- 既存の image モード（1〜9, BMP, ステレオ）に回帰がないこと。
